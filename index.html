1. Visão geral

Claude Code é uma ferramenta de desenvolvimento assistida por IA da Anthropic, criada para trabalhar diretamente com projetos de software. Ele atua como um agente capaz de ler arquivos, entender o contexto do repositório, editar código, executar comandos e ajudar em fluxos reais de desenvolvimento.

📖 Entende o projetoLê arquivos, pastas, diffs, logs e instruções do projeto.
✍️ Edita códigoCria, modifica, refatora e documenta arquivos.
🧪 Executa comandosPode rodar testes, builds, linters e comandos Bash.
Resumo: Claude Code combina terminal, IDE, contexto do projeto e automação para acelerar tarefas de desenvolvimento com IA.

Quando usar

Objetivo Exemplo de prompt
Entender um projeto Explique a arquitetura deste repositório.
Corrigir bug Investigue por que o checkout retorna erro 500.
Criar testes Crie testes para este service seguindo o padrão atual.
Revisar código Revise as mudanças atuais como code review.

2. Como instalar no VS Code

A extensão do Claude Code permite usar a ferramenta integrada ao editor, com acesso ao arquivo aberto, seleções de código e comandos da IDE.

Instalação pela aba de extensões

  1. Abra o VS Code.

    Use o editor já aberto na pasta do projeto ou abra uma nova janela.

  2. Abra a aba de extensões.

    No macOS, use Cmd + Shift + X. No Windows/Linux, use Ctrl + Shift + X.

  3. Pesquise pela extensão.

    Digite Claude Code no campo de busca.

  4. Instale a extensão oficial.

    Confirme que a extensão é da Anthropic antes de instalar.

  5. Autentique sua conta.

    Abra um projeto e siga o fluxo de login quando o VS Code solicitar.

5. Modelos: Claude, Haiku, Sonnet e Opus

Claude é a família de modelos. Dentro dela, existem modelos com diferentes equilíbrios entre velocidade, custo, qualidade e profundidade.

Modelo Perfil Melhor para Trade-off
⚡ Haiku Rápido e econômico Resumos simples, classificação, alto volume Menos profundo
🎼 Sonnet Equilibrado Coding diário, testes, docs, refatoração Melhor padrão geral
🧠 Opus Mais poderoso Arquitetura, bugs difíceis, planejamento complexo Mais custo/latência

Selecionar modelo

claude --model sonnet
claude --model opus
Regra prática: use Sonnet como padrão, Opus para tarefas complexas e Haiku para tarefas simples e rápidas.

6. Modes: Plan Mode, Auto-accept Edits e Auto Mode

Modes controlam o nível de autonomia do Claude Code durante uma sessão: quando deve apenas planejar, quando pode editar arquivos automaticamente e quando pode usar aprovações mais automáticas.

🧭 Plan Mode

Claude lê, investiga e propõe um plano antes de alterar arquivos ou executar ações de impacto.

✍️ Auto-accept Edits

Edições de arquivos podem ser aceitas automaticamente, reduzindo confirmações repetitivas.

🤖 Auto Mode

Tenta aprovar ações de baixo risco e pede confirmação para ações mais sensíveis.

Comparativo rápido

Mode O que faz Quando usar Controle
Plan Mode Analisa e cria plano antes de implementar. Arquitetura, refatorações grandes e tarefas com risco. Mais seguro
Auto-accept Edits Aceita edições automaticamente. Ajustes visuais, mudanças repetitivas e escopo claro. Intermediário
Auto Mode Automatiza aprovações de baixo risco. Fluxos longos em projetos versionados e confiáveis. Mais autônomo

Como mudar entre modes

Terminal / CLI 
Shift + Tab

Alterna entre modes disponíveis durante a sessão.

VS Code / Desktop

Use o seletor de mode na interface da sessão, próximo ao prompt ou permissões.

Atalhos e comandos úteis

/plan
claude --permission-mode plan
claude --enable-auto-mode
Boa prática: comece em Plan Mode quando a tarefa for grande ou incerta. Depois de aprovar o plano, mude para modos mais autônomos.

Comandos são instruções iniciadas com /. Eles controlam a sessão, consultam status, alteram configurações e executam workflows.

Comando

/btw qual comando lista branches no Git?
✅ Use para

Dúvidas rápidas, conceitos, comandos pequenos ou perguntas paralelas.

❌ Evite para

Decisões que precisam influenciar a tarefa principal.

10. Como criar comandos personalizados

No Claude Code, comandos personalizados podem ser criados por meio de skills. Uma skill é um arquivo Markdown com instruções reutilizáveis, que pode ser chamada como um comando.

Estrutura recomendada

.claude/
└── skills/
    └── summarize-changes/
        └── SKILL.md

Criar a pasta

mkdir -p .claude/skills/summarize-changes

Exemplo de SKILL.md

---
name: summarize-changes
description: Resume as mudanças não commitadas e aponta riscos.
---

# Summarize Changes

## Mudanças atuais

!`git diff HEAD`

## Instruções

Resuma as mudanças em até 5 bullets.

Depois liste:
1. riscos;
2. arquivos importantes;
3. sugestões de teste;
4. sugestão de mensagem de commit.

Chamar a skill

/summarize-changes

11. Skills

Uma skill é um pacote reutilizável de instruções, arquivos e, opcionalmente, scripts que ensina o Claude Code a executar uma tarefa específica seguindo um processo padronizado.

Em uma frase: skill é um workflow reutilizável. Você cria uma vez e chama sempre que precisar repetir aquele processo com consistência.
📋 Processo repetívelTransforma tarefas recorrentes em um fluxo documentado e fácil de reutilizar.
🎯 Instruções específicasDefine exatamente como Claude deve agir, validar e entregar o resultado.
🧩 Recursos extrasPode incluir referências, templates, assets e scripts auxiliares.

O que é uma skill?

Uma skill é como um pequeno “manual operacional” para o Claude. Ela descreve quando deve ser usada, quais passos seguir, quais padrões respeitar e qual formato de saída entregar. Isso evita repetir prompts longos toda vez que você precisar executar a mesma tarefa.

Quando utilizar skills?

Situação Por que usar uma skill?
Tarefa recorrente Você quer repetir o mesmo processo várias vezes sem reescrever o prompt.
Workflow com vários passos A skill guarda a ordem correta das etapas e reduz esquecimento.
Padrão do time Garante que revisões, documentos, testes ou relatórios sigam o mesmo formato.
Uso de templates A skill pode carregar modelos de saída, exemplos e referências.
Automação com scripts Quando há tarefas determinísticas, a skill pode incluir scripts para executar.

Skill vs agente

Skill

Melhor para processos, checklists, templates e workflows reutilizáveis.

Agente

Melhor para papéis especializados, como revisor, P.O., test writer ou analista de segurança.

Como criar uma skill

No Claude Code, uma skill normalmente fica dentro de .claude/skills/, com um arquivo principal chamado SKILL.md. Esse arquivo contém a descrição da skill e as instruções que Claude deve seguir. 

.claude/
└── skills/
    └── nome-da-skill/
        └── SKILL.md

Passo a passo rápido

  1. Escolha uma tarefa recorrente.

    Procure algo que você faz várias vezes e que vale a pena padronizar.

  2. Dê um nome curto para a skill.

    Exemplos: review-pr, generate-tests ou summarize-changes.

  3. Crie a pasta da skill.

    Ela deve ficar dentro de .claude/skills/.

  4. Crie o arquivo principal.

    O arquivo de entrada deve se chamar SKILL.md.

  5. Escreva uma descrição clara.

    Explique quando Claude deve usar essa skill e qual problema ela resolve.

  6. Adicione instruções e formato de saída.

    Inclua regras, passos, exemplos e o padrão esperado da resposta.

  7. Teste em uma tarefa real.

    Use a skill, observe o resultado e ajuste até o comportamento ficar previsível.

Comandos para criar a estrutura

mkdir -p .claude/skills/review-pr
touch .claude/skills/review-pr/SKILL.md

Exemplo de skill

---
name: review-pr
description: Revisa mudanças de código antes de commit ou pull request. Use quando o usuário pedir revisão de diff, revisão de PR, análise de riscos, sugestões de melhoria ou checklist antes de commitar.
---

# Review PR

Você é responsável por revisar as mudanças atuais do projeto como se estivesse fazendo um code review.

## Contexto inicial

Use o diff atual como base da análise:

!`git diff HEAD`

## Processo

1. Entenda o objetivo das mudanças.
2. Identifique arquivos mais importantes.
3. Procure bugs, riscos, inconsistências e problemas de manutenção.
4. Verifique se faltam testes.
5. Sugira comandos de validação.
6. Sugira uma mensagem de commit.

## Saída esperada

Use este formato:

### Resumo
- O que mudou em até 5 bullets.

### Riscos encontrados
| Severidade | Arquivo | Problema | Sugestão |
|---|---|---|---|

### Testes sugeridos
- Lista de testes ou comandos recomendados.

### Mensagem de commit sugerida
`tipo: descrição curta`

Como usar a skill

Depois de criada, você pode chamar a skill pelo nome dentro do Claude Code:

/review-pr

Ou pedir em linguagem natural:

Use a skill review-pr para revisar minhas mudanças atuais.
Boa prática: mantenha skills pequenas e específicas. Uma skill focada em “revisar PR” tende a funcionar melhor do que uma skill genérica que tenta revisar, testar, documentar e deployar tudo ao mesmo tempo.

12. CLAUDE.md

O CLAUDE.md é um arquivo Markdown de instruções que fica no projeto e é carregado pelo Claude Code no início das sessões. Ele serve para explicar como Claude deve trabalhar naquele repositório: stack, comandos, padrões, convenções, arquitetura, regras de segurança e preferências do time.

Em uma frase: o CLAUDE.md é a memória do Claude dentro do projeto — um lugar para registrar instruções persistentes que não precisam ser repetidas a cada conversa.
🧠 Memória do projetoGuarda padrões, decisões e instruções que Claude deve lembrar em novas sessões.
📐 Convenções do timeDefine estilo de código, estrutura de pastas, comandos de teste, build e lint.
🛡️ GuardrailsRegistra o que Claude deve ou não fazer, como evitar secrets, comandos destrutivos ou mudanças fora de escopo.

Por que ele é importante?

Sem um CLAUDE.md, Claude precisa descobrir ou perguntar várias coisas toda vez: como rodar o projeto, qual padrão de componentes usar, como nomear arquivos, quais comandos validar e quais decisões já foram tomadas. Com o arquivo, essas instruções viram contexto fixo do projeto.

Sem CLAUDE.md Com CLAUDE.md
Claude precisa inferir padrões pelo código. Claude já começa com regras e convenções explícitas.
Você repete comandos e instruções em toda sessão. Comandos importantes ficam documentados uma vez.
Maior chance de decisões inconsistentes. Maior consistência entre sessões e pessoas do time.

Equivalente ao AGENTS.md

Em outras ferramentas de agentes de código, é comum existir um arquivo chamado AGENTS.md para orientar como o agente deve trabalhar no projeto. No Claude Code, o equivalente mais importante é o CLAUDE.md.

Adendo: pense no CLAUDE.md como o “manual do projeto para o Claude”. Ele cumpre um papel parecido com AGENTS.md, .cursorrules ou arquivos de instruções usados por outras ferramentas, mas seguindo a convenção do Claude Code.

Onde criar?

O local mais comum é na raiz do projeto: 

meu-projeto/
├── CLAUDE.md
├── package.json
├── src/
└── README.md

Você também pode ter instruções mais específicas em subpastas quando um módulo precisa de regras próprias.

Como criar com /init

Dentro de uma sessão do Claude Code, use /init. Esse comando analisa o projeto e cria uma proposta inicial de CLAUDE.md com informações como comandos de build, testes, stack e convenções detectadas. 

# 1. Entre na pasta do projeto
cd meu-projeto

# 2. Abra o Claude Code
claude

# 3. Dentro da sessão, rode
/init
Boa prática: use o /init como ponto de partida, mas revise o arquivo manualmente. O melhor CLAUDE.md combina o que Claude detecta no código com decisões humanas do time.

O que colocar em um CLAUDE.md?

Seção Exemplo de conteúdo
Stack Next.js, TypeScript, Tailwind, Node, banco de dados e bibliotecas principais.
Comandos npm run dev, npm test, npm run lint, npm run build.
Padrões Estrutura de pastas, padrão de componentes, services, hooks, nomes e imports.
Regras Não alterar API pública sem pedir, não commitar secrets, pedir plano antes de refatorações grandes.
Fluxos Como criar feature, testar, revisar diff e preparar commit.

Template simples

# CLAUDE.md

## Projeto
Este projeto é um app Next.js com TypeScript e Tailwind CSS.

## Comandos principais
- Instalar dependências: npm install
- Rodar localmente: npm run dev
- Rodar lint: npm run lint
- Rodar build: npm run build
- Rodar testes: npm test

## Padrões de código
- Use TypeScript em modo strict.
- Prefira componentes pequenos e reutilizáveis.
- Mantenha lógica de negócio em src/lib.
- Mantenha componentes visuais em src/components.

## Regras para Claude
- Antes de mudanças grandes, proponha um plano.
- Não altere contratos públicos sem pedir confirmação.
- Não adicione dependências novas sem justificar.
- Nunca exponha tokens, chaves ou secrets.
- Depois de editar, sugira comandos de validação.

13. Agentes

Um agente é um especialista configurado para executar uma missão específica dentro do Claude Code. Em vez de pedir tudo para o Claude principal, você pode criar agentes com papéis claros, como revisor de código, criador de testes, documentador, analista de segurança ou especialista em performance.

Em uma frase: criar um agente é definir uma missão, um contexto de atuação e um conjunto de regras para que Claude delegue uma tarefa a um especialista.
🎯 Missão claraO agente sabe exatamente qual problema deve resolver e qual saída entregar.
🧠 Contexto próprioSubagentes podem trabalhar com seu próprio contexto, mantendo a conversa principal mais limpa.
⚡ ReusoVocê cria uma vez e reutiliza sempre que aquele tipo de tarefa aparecer.

O que é um agente?

No Claude Code, agentes personalizados são geralmente chamados de subagents. Eles são definidos em arquivos Markdown com um frontmatter em YAML e instruções detalhadas. Claude pode usar esses agentes para delegar tarefas específicas, especialmente quando a tarefa pede um perfil especializado ou quando faz sentido isolar a análise.

Vantagens de criar um agente

Vantagem Por que ajuda?
Define uma missão O agente tem um papel explícito, como “revisar segurança” ou “gerar testes”, evitando prompts genéricos.
Padroniza entregas Você pode definir formato de resposta, checklist, critérios de qualidade e limites de atuação.
Reduz repetição Não é preciso repetir sempre o mesmo prompt longo para tarefas recorrentes.
Melhora foco O agente trabalha como especialista em uma tarefa, sem tentar resolver tudo ao mesmo tempo.
Protege o contexto principal Tarefas de exploração, revisão ou análise podem ser delegadas sem poluir tanto a conversa principal.

Quando criar um agente?

✅ Crie quando...

A tarefa se repete, exige um papel especializado, precisa de checklist fixo ou tem um formato de saída bem definido.

❌ Evite quando...

A tarefa é única, simples, muito aberta ou ainda não tem processo claro. Nesse caso, um prompt comum costuma bastar.

Exemplos de bons agentes

Agente Missão Quando usar
code-reviewer Revisar mudanças procurando bugs, inconsistências e riscos. Antes de abrir PR ou fazer commit.
test-writer Criar testes seguindo padrões existentes do projeto. Depois de criar ou alterar uma feature.
security-reviewer Procurar riscos de segurança, secrets, validação e autorização. Antes de publicar fluxos sensíveis.
docs-writer Transformar implementação em documentação clara. Após finalizar uma feature ou módulo.

Guia rápido para criar um agente

  1. Defina a missão: o que esse agente deve resolver?
  2. Defina o escopo: onde ele pode atuar e o que deve evitar?
  3. Defina as ferramentas: ele pode ler arquivos, editar, rodar Bash ou apenas analisar?
  4. Defina o formato de saída: tabela, checklist, plano, diff, resumo ou relatório.
  5. Teste com uma tarefa real e ajuste as instruções até ficar previsível.

Criar um agente com /agents

A forma mais simples de começar é abrir a interface de agentes dentro do Claude Code com /agents. A partir dela, você pode criar um novo agente, escolher se ele será pessoal ou do projeto, definir nome, descrição, ferramentas e instruções. 

# 1. Entre na pasta do projeto
cd meu-projeto

# 2. Abra o Claude Code
claude

# 3. Dentro da sessão, abra a interface de agentes
/agents

Fluxo recomendado no /agents

  1. Rode /agents dentro do Claude Code.
  2. Escolha Create new agent.
  3. Escolha o local: Project para versionar no repositório ou Personal para usar em vários projetos.
  4. Dê um nome curto e claro, como code-reviewer ou test-writer.
  5. Escreva uma descrição dizendo quando Claude deve usar esse agente.
  6. Defina as instruções da missão, critérios de qualidade e formato de resposta.
  7. Salve e teste chamando Claude para usar esse agente em uma tarefa real.

Onde os agentes ficam?

Agentes podem ser criados como arquivos Markdown em pastas específicas. Use agentes de projeto quando a configuração deve ser compartilhada com o time, e agentes pessoais quando forem preferências suas. 

# Agente do projeto, versionado no repositório
.claude/agents/nome-do-agente.md

# Agente pessoal, disponível em vários projetos
~/.claude/agents/nome-do-agente.md

Template simples de agente

---
name: code-reviewer
description: Use este agente para revisar mudanças de código antes de commit ou PR.
tools: Read, Grep, Glob, Bash
---

# Code Reviewer

Você é um revisor de código cuidadoso.

## Missão
Revisar mudanças procurando bugs, riscos, inconsistências, problemas de legibilidade e falta de testes.

## Processo
1. Entenda o objetivo da mudança.
2. Analise os arquivos alterados.
3. Procure problemas de comportamento, segurança e manutenção.
4. Sugira melhorias práticas.
5. Indique comandos de validação quando fizer sentido.

## Saída esperada
Use este formato:

| Severidade | Arquivo | Problema | Sugestão |
|---|---|---|---|

Finalize com um resumo curto e uma lista de próximos passos.

Como pedir para Claude usar um agente

Use o agente code-reviewer para revisar as mudanças atuais.

Use o agente test-writer para criar testes para @src/lib/cart.ts.

Delegue para um agente de segurança revisar esse fluxo de autenticação.
Boa prática: agentes funcionam melhor quando têm uma missão pequena e objetiva. Evite criar um “super agente” que faz tudo; prefira vários agentes especialistas.

14. Prompt de agente de P.O.

Prompt pronto para criar um agente de Product Owner do projeto Mandáí, responsável por transformar ideias e requisitos em user stories claras, estruturadas e prontas para implementação.

Como usar: copie o prompt abaixo e use como instrução base ao criar um agente via /agents, ou salve como arquivo em .claude/agents/product-owner.md. 
Você é o Product Owner do projeto Mandáí — um app de delivery em construção como demonstração na série de mentorias DIO PRO.

Sua função é escrever user stories claras, estruturadas e prontas pra serem implementadas pelo time de desenvolvimento.

PADRÃO OBRIGATÓRIO PARA TODA USER STORY:

# US-XX: [Título da feature]

**Como** [tipo de usuário],
**eu quero** [ação ou funcionalidade],
**para que** [benefício ou valor de negócio].

## Critérios de aceite

- [ ] Critério 1
- [ ] Critério 2
- [ ] Critério 3

## Notas técnicas

- Detalhes relevantes pra implementação (componentes, dados, integrações)

## Fora de escopo

- O que NÃO faz parte dessa user story (pra evitar scope creep)

ONDE SALVAR:

Salve cada user story como um arquivo markdown em `/docs/user-stories/`, nomeado como `US-XX-<slug-do-titulo>.md` (numere sequencialmente a partir de US-01). Crie o diretório se ainda não existir.

REGRAS:

15. Contexto

Contexto é o conjunto de informações que Claude Code consegue considerar enquanto trabalha com você em uma sessão. Ele funciona como a “memória de trabalho” da conversa.

💬 ConversaPrompts, respostas, decisões e instruções dadas durante a sessão.
📁 ProjetoArquivos lidos, trechos de código, diffs, logs e estruturas do repositório.
🛠️ ExecuçõesResultados de comandos, testes, builds e ferramentas usadas pelo Claude Code.
Ideia principal: quanto melhor o contexto, melhores tendem a ser as respostas. Mas contexto demais, bagunçado ou antigo pode atrapalhar a precisão.

O que são turnos?

Um turno é cada troca dentro da conversa. Quando você envia uma mensagem, isso é um turno do usuário. Quando Claude responde, isso é um turno do assistente. Em uma sessão longa, o contexto é formado por uma sequência desses turnos, somada aos arquivos lidos, comandos executados e resultados retornados pelas ferramentas.

👤 Turno do usuário

É quando você envia uma instrução, pergunta, correção, arquivo ou decisão para Claude Code.

🤖 Turno do Claude

É quando Claude responde, propõe plano, edita arquivos, pede confirmação ou interpreta resultados.

Como os turnos funcionam na prática?

Claude Code usa os turnos anteriores para entender o objetivo atual. Por isso, se você pediu um plano, aprovou uma abordagem e depois pediu a implementação, Claude tenta considerar essa sequência como parte do mesmo trabalho.

Turno Exemplo Como afeta o contexto
1. Usuário Analise o fluxo de autenticação. Define o objetivo inicial.
2. Claude Explica arquivos, riscos e propõe plano. Adiciona hipóteses e decisões temporárias.
3. Usuário Implemente apenas a etapa 1. Refina o escopo e limita a ação.
4. Claude Edita arquivos e roda testes. Adiciona mudanças, resultados e próximos passos.
Importante: quanto mais turnos a sessão acumula, mais informações entram na memória de trabalho. Isso ajuda na continuidade, mas também pode trazer ruído se a conversa mudar muito de direção.

O que entra no contexto?

Fonte Exemplo Impacto
Mensagens da conversa Objetivo, restrições, decisões e próximos passos. Ajuda Claude a manter continuidade.
Arquivos do projeto @src/app/page.tsx, configs, componentes e services. Ajuda Claude a entender a implementação real.
Comandos executados npm test, npm run build, git diff. Ajuda Claude a reagir a erros e validar mudanças.
Instruções do projeto CLAUDE.md, padrões de código e regras internas. Ajuda Claude a seguir convenções.

O que é o comando /context?

O /context mostra informações sobre o uso do contexto atual da sessão. Ele é útil para entender se a conversa está ficando grande demais, se muitos arquivos foram carregados ou se vale a pena compactar antes de continuar. 

/context

Quando usar /context?

Antes de uma tarefa grande

Verifique se a sessão ainda tem espaço e clareza para uma nova etapa.

Depois de muitos arquivos/logs

Confira se a conversa não ficou carregada demais com informações antigas.

Antes de refatorações

Garanta que Claude ainda tem o contexto correto do plano e das decisões.

Quando a resposta piorar

Se Claude parecer confuso, o contexto pode estar poluído ou longo demais.

Como limpar o contexto

Para começar uma conversa nova, use /clear. Isso limpa o histórico da sessão atual e ajuda quando você quer mudar totalmente de assunto ou recomeçar com um objetivo novo. 

/clear
Use /clear quando: terminou uma tarefa, vai começar outra sem relação, o Claude ficou preso em uma hipótese errada ou você quer evitar que decisões antigas influenciem o próximo passo.

Como compactar o contexto

Para manter a continuidade sem carregar todo o histórico detalhado, use /compact. Ele resume a conversa, preservando as partes importantes para seguir trabalhando. 

/compact

Também dá para orientar o que deve ser preservado:

/compact preserve decisões técnicas, arquivos alterados, bugs encontrados e próximos passos

Quando compactar?

Situação O que fazer
A conversa ficou longa, mas a tarefa continua. Use /compact.
Você já decidiu a arquitetura e vai iniciar implementação. Use /compact preserve plano aprovado e decisões.
Muitos logs ou erros antigos estão no histórico. Use /compact para reduzir ruído.
Vai mudar totalmente de tarefa. Prefira /clear.

Regra prática

Compacte quando quiser continuar a mesma tarefa com menos ruído. Limpe quando quiser começar uma tarefa nova do zero.

19. Git cheat sheet: primeiro commit

Guia rápido com os comandos essenciais para versionar e publicar seu primeiro projeto no GitHub.

Objetivo: sair de uma pasta local com código para um repositório Git com primeiro commit enviado para o GitHub.

Fluxo completo do zero

# 1. Entrar na pasta do projeto
cd meu-projeto

# 2. Iniciar o Git na pasta
git init

# 3. Verificar o status dos arquivos
git status

# 4. Adicionar todos os arquivos ao stage
git add .

# 5. Criar o primeiro commit
git commit -m "first commit"

# 6. Renomear a branch principal para main
git branch -M main

# 7. Conectar com o repositório remoto do GitHub
git remote add origin https://github.com/seu-usuario/seu-repositorio.git

# 8. Enviar o projeto para o GitHub
git push -u origin main

Comandos essenciais

Comando Para que serve
git init Inicializa um repositório Git na pasta atual.
git status Mostra arquivos modificados, adicionados e pendentes.
git add . Adiciona todos os arquivos alterados ao stage.
git commit -m "mensagem" Salva um snapshot das mudanças com uma mensagem.
git log --oneline Lista commits de forma resumida.
git remote -v Mostra os repositórios remotos configurados.
git push Envia commits locais para o GitHub.

Configuração inicial do Git

Use uma vez na máquina para configurar seu nome e e-mail nos commits.

git config --global user.name "Seu Nome"
git config --global user.email "seu-email@example.com"

# Conferir configuração atual
git config --global --list

Criar um arquivo .gitignore

Evite versionar dependências, builds, variáveis de ambiente e arquivos locais.

# Exemplo para projetos Node/Next.js
node_modules
.next
dist
build
.env
.env.local
.DS_Store

Erros comuns e como resolver

Erro: remote origin already exists

O remoto origin já existe. Confira ou altere a URL.

git remote -v
git remote set-url origin https://github.com/seu-usuario/seu-repositorio.git
Erro: src refspec main does not match any

Normalmente acontece quando ainda não existe commit ou a branch tem outro nome.

git status
git add .
git commit -m "first commit"
git branch -M main
git push -u origin main
Quero desfazer um arquivo antes do commit 
# Remover arquivo do stage, mas manter alteração no arquivo
git restore --staged nome-do-arquivo

# Descartar alterações locais de um arquivo
git restore nome-do-arquivo
